01 — Three Primitives

The building blocks

Claude Code's extension system is built on three composable primitives. Each solves a different problem: who does the work, how they do it, and what they can interact with.

◉

Agents

The "who"

Autonomous Claude sessions with a defined persona, role, and set of capabilities. Each agent has a specific job — design, code, evaluate, write copy — and carries its own instructions, tool access, and model preference.

Defined .claude/agents/*.md
Spawned By orchestrator via Task tool
Scope Own context window, own tools
⚙

Skills

The "how"

Reusable workflow instructions that define a multi-phase process. Skills are methodology — they describe what steps to follow, what artifacts to produce, and what validation to run. They load as context into whatever agent executes them.

Defined .claude/skills/*/SKILL.md
Invoked Via /skill-name or by agents
Scope Loaded into executing agent's context
⚒

Tools

The "what"

Concrete actions an agent can perform — reading files, writing code, running commands, searching codebases, making web requests. Tools are the interface between Claude's reasoning and the real world.

Defined Built into Claude Code + MCP servers
Access Per-agent tool allowlist in frontmatter
Scope Single action, returns result
02 — Agents

Specialized personas

An agent is a Claude session with a defined identity. The YAML frontmatter in its .md file declares its name, description, model, available tools, and loaded skills. The markdown body provides its instructions — role, team behavior, context loading, validation criteria, and guiding principles.

Agent Definition Structure
--- YAML frontmatter --- name: brand-designer description: "Translates brand identity into design systems..." tools: Read, Write, Grep, Glob, WebSearch model: opus memory: project skills: - design-system - frontend-design - theme-factory --- # Markdown body = agent instructions ## Role ← What this agent does ## Team Role ← How it interacts with other agents ## Context ← What docs to read before starting ## Validation ← Quality gates the agent must pass ## Principles ← Decision-making guidelines

03 — Skills

Reusable workflows

A skill is a SKILL.md file that defines a multi-step process. Skills don't execute on their own — they load as instructions into the agent that runs them. A skill can orchestrate multiple agents, define phases, specify artifacts, and include validation checks. Users invoke skills with /skill-name.

Skill Definition Structure
--- YAML frontmatter --- name: creative-studio description: "Orchestrates 3 parallel creative teams..." --- # Markdown body = workflow definition ## Input ← What arguments the skill accepts ## Workflow ← Phase-by-phase execution steps Phase 1: Design (spawn 3 brand-designer agents) Phase 2: Copy (spawn 3 copywriter agents) Phase 3: Build (spawn 3 expert-developer agents) Phase 4: Review (spawn 1 customer agent) Phase 5: Iterate or Ship ## Rules ← Hard constraints during execution ## References ← Supporting files (templates, prompts)

Two types of skills

Methodology Skills

Load as context

Provide instructions and guidelines that shape how an agent works. They don't spawn sub-agents — they change the executing agent's behavior by adding constraints, checklists, and quality standards.

Examples frontend-design, brainstorming, code-style
Agents 0 spawned — loaded into executor

Orchestration Skills

Coordinate agents

Define multi-agent workflows that spawn specialized sub-agents in parallel or sequence. They control the flow — what agents to spawn, when, with what instructions, and how to synthesize results.

Examples creative-studio, agent-team, feature-frontend
Agents Multiple spawned per phase

04 — Tools

Concrete actions

Tools are the hands and eyes of an agent. Each tool performs one specific action — read a file, run a command, search code, edit text. Agents declare which tools they can access in their frontmatter. The orchestrator decides which tools to grant each sub-agent based on its role.

File Operations
Read — View file contents, images, PDFs
Write — Create or overwrite files
Edit — Precise string replacements
Glob — Find files by pattern
Grep — Search file contents (regex)
Execution
Bash — Run shell commands (git, npm)
Task — Spawn sub-agent for complex work
Skill — Invoke a skill by name
NotebookEdit — Edit Jupyter notebooks
Web & Interaction
WebSearch — Search the web for info
WebFetch — Fetch and process URLs
AskUserQuestion — Interactive prompts
MCP Tools — External server integrations
Tool Access Control
// Agent frontmatter restricts which tools each agent can use brand-designer: Read, Write, Grep, Glob, WebSearch // No Bash, no Edit expert-developer: Read, Write, Edit, Grep, Glob, Bash // Full code access customer: Read, Grep, Glob // Read-only evaluator copywriter: (all tools via orchestrator) // Full access for content // The orchestrator (Claude) has access to ALL tools // Sub-agents get restricted based on role

Principle of Least Privilege

Agents only get the tools they need. A customer evaluator has no Write or Bash — it can only read and analyze, preventing accidental modifications.

Hooks Extend Tools

Agents can define hooks that fire after tool use. The expert-developer auto-formats code after every Edit or Write call via a PostToolUse hook.

06 — Orchestration

Execution patterns

How the orchestrator coordinates agents depends on the skill being executed. Three patterns emerge: single-agent (one specialist does it all), pipeline (agents execute in sequence), and parallel teams (multiple agents compete or collaborate simultaneously).

Single Agent

Linear execution

Orchestrator loads a methodology skill and executes all phases itself. No sub-agents spawned.

User → Orchestrator + skill → Output
Used by: frontend-design, brainstorming

Pipeline

Sequential agents

Agents execute in sequence, each consuming the previous agent's output. Orchestrator manages handoffs.

A → artifact → B → artifact → C
Used by: design-system, feature-frontend

Parallel Teams

Competing agents

Multiple agents work the same brief simultaneously from different angles. An evaluator picks the winner.

A \ B —→ evaluator → winner iterates C /
Used by: creative-studio, agent-team
07 — Live Example

Creative Studio in action

The creative-studio skill demonstrates all three primitives working together. A user types /creative-studio "landing page" and the system orchestrates 10+ agent spawns across 5 phases.

0

Prepare Brief

Orchestrator reads project docs, selects 3 creative angles, writes structured brief to disk. No agents spawned — the orchestrator does this using its own tools.

skill: creative-studio Read Write
1

Design (3 teams parallel)

Spawns 3 brand-designer agents simultaneously via 3 Task calls in a single message. Each reads the brief with a different creative angle and produces a design specification. Each carries the frontend-design methodology skill.

agent: brand-designer ×3 skill: frontend-design Task Read Write
2

Copy (3 teams parallel)

Spawns 3 copywriter agents. Each reads their team's design spec and writes all copy — headlines, body, CTAs, microcopy. The copy matches the creative angle.

agent: copywriter ×3 Task Read Write
3

Build (3 teams parallel)

Spawns 3 expert-developer agents. Each reads the design spec + copy and implements production-ready code. The PostToolUse hook auto-formats on every write.

agent: expert-developer ×3 Task Write Edit Bash
4

Customer Review

Spawns 1 customer agent. Reads all 3 team outputs, scores each across 5 dimensions, picks the winner, provides actionable feedback. If average score < 8, the winning team iterates (loop back to Phase 1 with 1 team).

agent: customer ×1 Read Grep Glob

08 — File Structure

Where everything lives

All configuration lives under .claude/ in the project root. This directory is version-controlled and shared across the team. CLAUDE.md at the root provides project-level instructions that all agents inherit.

project/ CLAUDE.md # Project-level instructions (always loaded) .claude/ agents/ # Agent definitions brand-designer.md # YAML frontmatter + instructions expert-developer.md customer.md copywriter.md skills/ # Skill definitions brainstorming/ SKILL.md # Workflow definition creative-studio/ SKILL.md references/ # Supporting files creative-angles.md brief-template.md team-handoff-prompts.md design-system/ SKILL.md references/ token-generation-guide.md design-imperatives.md desktop-patterns.md mobile-patterns.md frontend-design/ SKILL.md agent-team/ SKILL.md references/ team-patterns.md conflict-avoidance.md rules/ # Additional rules (always loaded) coding-rules.md docs/ DESIGN/ DESIGN_SYSTEM.md # Generated by design-system skill showcase.html # Visual validation artifact SCREENS.md # Screen-by-screen specs

10 — Takeaways

Key insights

The mental model for building effective Claude Code workflows.

Agents are specialists, not generalists

Each agent has a narrow role with restricted tools. A brand-designer can't run shell commands. A customer can't modify files. Constraints produce better output than freedom.

Skills are the real orchestration layer

Skills define the workflow — what agents to spawn, in what order, with what inputs. The orchestrator follows the skill's instructions. Complex behavior emerges from simple skill composition.

Tools are the boundary with reality

Everything an agent "does" happens through a tool call. Reading a file, writing code, running tests, spawning another agent — all tool calls. The tool layer is where reasoning meets the real world.

Artifacts are the handoff mechanism

Agents communicate by writing files. The brand-designer writes design-spec.md. The copywriter reads it and writes copy.md. The expert-developer reads both and writes code. Files are the shared memory.

Parallel teams beat sequential refinement

Three teams competing on the same brief with different angles produces better results than one team iterating. Competition surfaces ideas that consensus misses.

CLAUDE.md is the constitution

Every agent inherits CLAUDE.md instructions. Project-level standards, quality gates, and philosophy flow to all sub-agents automatically. One source of truth for the whole system.

Plan first, team second

Wrong task decomposition costs 500k+ tokens. Always brainstorm and plan before spawning a team. Define shared contracts before any agent starts writing code.

Everything is markdown

Agents, skills, rules, design systems, specs — all defined in markdown files. No special runtime, no compilation, no deployment. Version-controlled, human-readable, composable.
